# パーソナル秘書アプリ 機能アイデア

作成日: 2026-07-12
元プラン: [decide-app-features](../plans/decide-app-features.md)

> **更新 2026-07-12**: Gmail / Google カレンダーの MCP 認証が完了し、実データ調査を反映した
> （調査結果の詳細は[プラン側](../plans/decide-app-features.md#調査結果)）。
> 残る未取得情報は Canvas の Access Token（または iCal URL）のみ。

**実データから得た主な補正点**:

1. カレンダーの予定は少ない（2.5 ヶ月で 9 件）が通院・学校などクリティカル → 「空き時間の把握」より **少数の重要予定を確実にリマインド** する方が価値が高い
2. 人間からのメールはほぼゼロ。トリアージの対象は **通知メールからのアクショナブル抽出**（サブスク期限・配送・学校・銀行）に絞る。「返信忘れ検知」は優先度を下げる
3. ユーザーは既に n8n（Autopilot）で毎朝ニュースダイジェストを自分宛てメール配信している → **朝ブリーフィング体験は実証済み**。ただし Gmail は自動配信で混んでいるため、秘書の通知はメール以外のチャネルが望ましい
4. Canvas の通知メールは採点とアナウンスのみで **締切はメールに流れない** → Canvas API / iCal が必須依存

## 1. 管理するもの（情報ドメイン）

秘書が扱う情報を「ドメイン × 秘書がやること」で整理する。
秘書の動作は基本的に **把握（収集）→ 整理（要約・構造化）→ 通知（プッシュ）→ リマインド → 代行** の 5 段階。まずは把握〜リマインドまでを対象にし、代行（メール返信の下書きなど）は後期フェーズとする。

### 1-1. 予定（Google カレンダー）

- 今日・今週の予定の把握と要約
- 予定の直前リマインド（移動時間を考慮できるとなお良い）
- 空き時間の把握（タスクの割り当て先として使う）— 実データでは予定が少ないため優先度低

実データ: 対象はメインカレンダー（akiraak@gmail.com）とパートナー共有の cessna。Family / WE:Topic は未使用。
予定は通院（Kaiser）・学校日程・定期購入が中心で件数は少なく、1 件の重みが大きい。

### 1-2. メール（Gmail）

- 定期チェックして **対応が必要なメールだけ** 抽出・要約（トリアージ）
- 通知系メール（Canvas、GitHub、請求など）の自動分類 → ドメイン別の情報源として再利用
- ~~返信忘れの検知~~ → 実データでは人間からのメールがほぼゼロのため見送り
- 将来: 返信ドラフトの生成（代行フェーズ）

実データに基づくトリアージ基準（初期ルール案）:

| 区分 | 例 | 扱い |
|---|---|---|
| 要対応 | サブスク期限（Google One 等）、銀行・支払い、学校事務（navigate@shoreline.edu）、セキュリティ警告（パスワード漏洩） | ブリーフィングで明示 + 期限があればリマインド |
| 参考 | Canvas 採点結果、Amazon 配送状況 | ブリーフィングに 1 行 |
| 無視 | プロモ（Amazon store-news、NordVPN）、ニュースレター（NYT、Mackerel、note） | 出さない |
| 除外 | 自分宛て自動送信（Autopilot ニュース、[Autopilot] レポート、セルフメモ） | 秘書の入力にしない（ただしセルフメモはタスクのキャプチャ源として将来利用の余地あり） |

### 1-3. 学校（ESL / Canvas）

- 課題・締切・アナウンスの取得と一覧化
- 締切前リマインド（例: 3日前・前日・当日朝）
- 授業スケジュールとカレンダーの統合
- 取得手段は **iCal フィード（認証不要・手軽）に確定**。REST API（Access Token）は
  **Shoreline 側で管理者が学生のトークン発行を無効化しており使えない**（2026-07 確認済み）。
  通知メールもフォールバックにならない（実データでは採点・アナウンスのみで **締切が流れてこない**）
- iCal で取れるのは締切・カレンダーイベントのみ。アナウンス要約や成績・提出状況は取得不可 →
  MVP スコープは「締切トラッカー」に限定する（アナウンス要約は当面スコープ外）

実データ: インスタンスは `https://shoreline.instructure.com`（Shoreline Community College）。
現コースは SU26「ESL/ELL for College and Career 4 & ESLAF 063」（course id 2742290、夏学期 7/1〜）。
残るユーザー作業は **iCal Calendar Feed の URL 取得**（Canvas → Calendar → 右下「Calendar Feed」）。

### 1-4. 開発プロジェクト（GitHub）

- 並行 4〜6 本のプロジェクトの横断把握: 最終更新、放置日数、各リポジトリの TODO.md の残タスク
- 「今週触っていないプロジェクト」の可視化（意図的な放置か忘却かの判断材料）
- CI 失敗・Dependabot アラートなどの通知集約

### 1-5. 自分のタスク（横断 TODO）

- カレンダーの予定・Canvas の締切・各リポジトリの TODO を **一つのタスクビュー** に統合
- 「今日やるべきこと」の自動提案（締切 × 空き時間から）
- これは独立ドメインというより 1-1〜1-4 の統合レイヤー

### 1-6. 家庭・生活（連携候補・優先度低）

- kitchen-living（在庫・買物リスト・レシピ・カレンダー）との連携: 買物リマインドをブリーフィングに混ぜる等
- g3plus サーバの死活・ディスク・稼働サービス（n8n / Wiki.js / esltext）の監視通知

### 1-7. ESL 学習（連携候補・優先度低）

- esl-learning-assistant / esl-text-audio と連動した予習リマインド・教材提示

## 2. ユーザーインターフェース（表示・通知）

### 2-1. 基本の考え方: プッシュ + プル の二本立て

| 型 | 役割 | 具体例 |
|---|---|---|
| **プッシュ（通知）** | 秘書側から能動的に届ける。見に行かなくても回る状態を作る | 朝のブリーフィング、締切リマインド、重要メール即時通知 |
| **プル（ダッシュボード / 対話）** | 詳細を見たい・聞きたいときにユーザーが見に行く | 統合タスクビュー、チャットで「今週の締切は?」 |

秘書アプリの価値の中心は **プッシュ側**（「開かなくても教えてくれる」）にあるので、MVP はプッシュから作る。

### 2-2. プッシュ通知の設計案

1. **朝のブリーフィング（毎朝 1 通）** — 最優先
   - 今日の予定 / 直近の Canvas 締切 / 要対応メール / 昨日の GitHub 活動サマリ
   - 情報を 1 通に集約することで通知疲れを防ぐ
2. **締切・予定リマインド（イベント駆動）**
   - Canvas 課題の 3日前 / 前日、予定の直前
3. **即時通知（例外のみ）**
   - 本当に緊急なものだけ（重要メール、サーバダウン等）。デフォルトは静か、が原則

**配信チャネル候補**（要ユーザー確認 — 普段一番見るものに届けるのが正解）:

| チャネル | 実装コスト | 特徴 |
|---|---|---|
| メール（Gmail 宛） | 低 | 確実だが「メールを減らす」目的と矛盾しがち。実データ上 Gmail は自動配信で混雑済み → 非推奨 |
| Discord / Slack Webhook | 低 | 実装最速。Bot 化すれば対話（プル）も同じ場所でできる。ボタンで「済/明日/詳細」の操作通知も可 |
| LINE (Messaging API) | 中 | 日本の生活動線に一番近い（ただし PC 動線から遠い） |
| スマホ push（ntfy / Pushover） | 低 | シンプル。対話はできない |
| **ネイティブ iOS アプリ + APNs** ← **採用** | 高 | 別項 2-2-1。UX 最強・プライバシー確保。MVP から採用と決定 |

いずれも **g3plus の n8n をディスパッチャ** にすれば、秘書アプリ本体は「n8n に投げる」だけで済み、チャネル変更にも強い。

#### 2-2-1. ネイティブ iOS アプリ通知の検証（2026-07 実施）

「iOS アプリを作り、その push で通知を受ける」構成の妥当性を検証した結果。

**技術的成立性: ○**。g3plus（Linux）→ APNs は HTTP/2 + JWT（`.p8` キー）で送信可能。送信元が自宅サーバでも問題ない。

**必要コスト（＝ハードル）**:

- **Apple Developer Program $99/年（必須）**。無料アカウントではリモートプッシュ不可
- Mac + Xcode（esl-learning-assistant で保有済みのはず）
- 自分用配布: 有料アカウント + TestFlight（90 日ビルド）。無料枠は 7 日で失効
- サーバ側の APNs 実装 + デバイストークン管理 + 証明書/プロビジョニング保守

**検証の核心**: 通知頻度は「朝 1 通/日 + たまに締切リマインド」で、到達性・性能の要求は皆無。
→ **ネイティブアプリの価値は「通知の配送」ではなく「UX」にしかない**。UX 価値は:

- ホーム/ロック画面ウィジェット・Live Activities / Dynamic Island（アンビエント表示、2-6 参照）
- アプリ内ダッシュボード（プル側）を同居できる
- **プライバシー**: カレンダーに医療情報（消化器科・エコー検査・発疹の結果電話など）が含まれる。
  Discord / Slack / ntfy.sh 経由は本文が第三者サーバを通る。自前 APNs なら
  「起動シグナルのみ push → 本文は g3plus から TLS 取得」で内容を自インフラに閉じられる
- ユーザーは iOS 開発者（Swift 資産あり）で学習コストが低い

**当初検証の結論（記録）**: 通知の配送だけを見ればオーバーキルで、MVP のつなぎには Discord 等の軽量チャネルが合理的。

**採用した方針（2026-07、ユーザー決定）**: 上記トレードオフを承知のうえで **最初からネイティブ iOS アプリで行く**。
理由は UX（ウィジェット / Live Activities / アプリ内ダッシュボード）とプライバシー（医療情報を自インフラに閉じる）を
初日から取りに行くこと、およびユーザーが iOS 開発者で Swift 資産を持ち追加学習コストが低いこと。
つなぎチャネル（Discord / Pushover / PWA）は採用しない。

これに伴う MVP 構成:

- **バックエンドはチャネル非依存の設計を維持**（コレクタ + LLM + ディスパッチャ）。当面の配信先は iOS アプリ 1 つだが、将来の追加に備える
- **push ペイロード方式**（要決定・下記）:
  1. **フル内容を payload に載せる** — 実装最短。ただし本文が APNs（Apple のサーバ）を通る
  2. **シグナルのみ push → アプリが g3plus から本文取得**（mutable-content / background fetch + 認証付き API）— 医療情報を自インフラに閉じられる。実装はやや増える
  - 推奨: v1 は方式 1 で早く動かし、方式 2 へ移行（プライバシーが動機なので早めに 2 へ寄せる）
- 配布は TestFlight（個人用、有料アカウント前提）

### 2-3. プル側（ダッシュボード / 対話）の設計案

- **Web ダッシュボード**: 統合タスクビュー + 各ドメインの詳細。g3plus 上で `secretary.chobi.me` のように運用（esltext と同じデプロイパターン）
- **チャット対話**: ブリーフィングと同じチャネル（Discord/Slack/LINE）で Bot に質問 → LLM が答える。「今週の締切は?」「昨日返信し忘れたメールある?」
- CLI は開発者的には楽だが「秘書」体験としては弱いので補助扱い

### 2-4. デバイス比較: PC vs スマホ

| | PC（Windows + WSL） | スマホ |
|---|---|---|
| 使う場面 | デスクで作業中・開発中 | 起床直後・移動中・外出先・就寝前 |
| プッシュ手段 | Windows トースト（ntfy デスクトップ、Discord/Slack デスクトップ） | OS プッシュ通知（LINE / Discord / ntfy / Pushover）、ロック画面 |
| プル手段 | Web ダッシュボード、CLI/TUI、ターミナル起動時表示 | チャット Bot、PWA、ホーム/ロック画面ウィジェット |
| 強み | 一覧性が高い。キーボードで即操作（返信下書きの承認等）。開発動線と同じ場所 | 常に携帯。朝一・外出中に届くのはスマホだけ。時刻ベース通知と相性が良い |
| 弱み | PC の前にいない時間に届かない。朝のブリーフィングの受け口には不適 | 長文・一覧に弱い。ネイティブアプリは開発コスト大 |
| 向く役割 | **プル（詳細確認・操作）** | **プッシュ（通知・ブリーフィング）** |

結論: どちらかを選ぶのではなく役割分担。**通知はスマホ、詳細と操作は PC**。
Discord / Slack のようなマルチデバイス同期チャネルを背骨にすれば、同じメッセージが両方に届き既読も同期されるため、この分担が追加実装なしで成立する（LINE は PC 動線から遠い、ntfy は一方向、という点で劣る）。

### 2-5. デバイス別の入力・編集の設計（何をさせ、何をさせないか）

**設計原則: マスターデータは元のサービスで編集し、秘書アプリでは編集させない。**
予定は Google カレンダー、メールは Gmail、課題は Canvas が正。秘書アプリに編集機能を持たせると二重管理・同期バグの温床になる。秘書アプリが受け付ける入力は次の 3 種に限定する:

1. **判断**: 済んだ / 後で / 無視 のワンタップ
2. **指示・質問**: 自然言語チャット
3. **キャプチャ**: 思いついたタスク・メモの投げ込み（整形は秘書側で行う）

デバイス別の割り当て:

| | スマホ | PC |
|---|---|---|
| させる（入力） | 通知上のワンタップ判断（済 / 明日再通知 / 無視）、チャット短文・音声入力でのキャプチャ、代行実行の承認ボタン | 設定・ルールの編集（トリアージ基準、通知時刻、チャネル）、返信下書きの修正、タスクの一括整理、長い指示 |
| させない | 長文の編集・執筆、設定変更、一括操作 | （PC 固有の禁止はほぼなし。共通原則のみ） |
| 理由 | 「秒で終わる操作」だけに絞ると通知が処理装置として機能する。編集 UI を作り込むほど使われなくなる | 間違えると通知全体が壊れる入力（設定・ルール）は、全体が見える広い画面に寄せる |

両デバイス共通でさせないこと:

- **マスターデータの編集**（上記原則）。秘書アプリが自前で持つ状態は「既読・済・スヌーズ」等のフラグ類のみ
- **承認なしの代行実行**: メール送信・イベント削除など外部に影響する操作は、必ず「下書き提示 → 明示的承認」を挟む。承認ボタンはスマホでも押せてよいが、内容の修正は PC に寄せる

副次効果: 編集機能を作らないことで MVP の実装量が大きく減る（読み取り + フラグ + チャットで成立する）。

### 2-6. その他の表示・通知アイデア

実装が軽い順に有望なもの:

1. **Google カレンダーへの逆書き込み**: 秘書が締切・今日の要点をカレンダーイベントとして書き込む。既存のカレンダー UI（全デバイス + ウィジェット）がそのまま表示面になり、UI 実装ゼロ。「Phase 0」候補
2. **iOS ロック画面 / ホーム画面ウィジェット**: 通知ではない常時表示（アンビエント）。Scriptable を使えば Swift 不要で、秘書 API の JSON を描画できる。締切カウントダウン等に最適
3. **音声ブリーフィング（TTS）**: 朝のブリーフィングを音声化し身支度中に聞く。esl-text-audio の音声合成資産を流用可能。英語版で読み上げれば ESL 練習との掛け算になる
4. **ターミナル / Claude Code 起動時ブリーフィング**: シェル起動時に予定・締切・放置プロジェクトを数行表示。開発者の一日はターミナルから始まるため、確実に目に入る
5. **アクション付き通知**: Discord Bot のボタンで「済んだ / 明日再通知 / 詳細」を通知上で完結。通知が「読むだけ」から「処理できる」に変わる

中期的な候補:

6. **PWA + Web Push**: 1 つの Web アプリで PC/スマホ両対応、通知も Web Push に統一。Phase 4 のダッシュボードの実装形態として有力
7. **常設ディスプレイ（E-ink / 古いタブレット）**: キッチン・机に置く常時表示。g3plus が API を配信し、kitchen-living（買物リスト）と同居させると家庭内ダッシュボードになる

### 2-7. アーキテクチャの示唆（UI 案からの逆算)

- コア: 各情報源のコレクタ（Calendar / Gmail / Canvas / GitHub） + 要約・判断を行う LLM 層 + 配信（n8n 経由）
- 定期実行: g3plus 上の cron または n8n スケジュール
- データは一度自前 DB（SQLite で十分）に正規化して溜めると、ダッシュボードと通知の両方から使い回せる

## 3. 優先度案（MVP → 拡張）

|---|---|---|

| Phase 5 | GitHub 横断把握、kitchen-living / g3plus 監視連携、メール返信代行 | 適宜 |
