## ナビゲーション構成の決定（2026-07-14 更新）

**下部ナビ4タブ: HOME / GitHub / Calendar / Setting**（ユーザー決定）。
HOME に **案A「統合フィード（優先順1画面）」** を据える。案B/Cは見送り。

- **HOME** = 横断ダイジェスト（案A）。締切・TODOサマリー・要対応・昨日のGitHub の要点を優先順に。
  各セクション見出しは対応タブへのリンク（「GitHub ›」「Calendar ›」）
- **GitHub** = 昨日の活動（リポジトリ別 commits/PR）＋各 `TODO.md` の次の作業＋放置検知
- **Calendar** = 週ストリップ＋アジェンダ。Google 予定 + Canvas 締切 + サブスク期限を1本の時系列に統合
- **Setting** = 通知・連携ソース・`TODO.md` 読み取り対象リポジトリ

役割分担: **要点は HOME、全量は各タブ**（同じデータの見せ方を変えるだけ。二重に作り込まない）。
毎朝の push は HOME の要約を届け、タップで HOME を開く。タブはプル側。

これに伴い **GitHub の作業把握を MVP に取り込む**（下記「よく使う機能」）。元計画では Phase 5 だが前倒し。

### よく使う機能（この設計が満たすもの）

1. 朝に **今日の作業と直近の締切** を確認する
2. **GitHub の前日までの作業** と **今後の作業（各リポジトリの `TODO.md`）** を確認する

## スコープ

MVP の画面 = **オンボーディング + ロック画面通知 + 4タブ本体（HOME / GitHub / Calendar / Setting）**。
初回の通知許可 → 毎朝の受け取り → HOME で確認 → 各タブで深掘り、を通す。
実装は **HOME を先に動かし、GitHub / Calendar タブは順に肉付け**する段取り（[MVP プラン](../plans/archive/mvp-morning-briefing.md)）。
ウィジェット / Live Activities / チャットは後フェーズ（[spec Phase 3+](app-features.md)）。

## 画面

### 1. オンボーディング（初回起動）
- 目的: 通知を許可し、デバイストークンを backend（`POST /devices`）に登録する
- 内容: アプリの説明を「カレンダー / Canvas 締切 / 要対応メール」の3点に絞る
- ボタン: 「通知を許可して始める」/「あとで」
- 許可が下りたら APNs のデバイストークンを取得 → backend に送信

### 2. ロック画面通知（毎朝 07:00 PT）
- **主役のチャネル。** v1 は要点をフル本文で push（[ペイロード方式1](../plans/archive/mvp-morning-briefing.md)）
- 文面例: 「タイムスケジュールの予定はなし。締切が2日後、要対応メール2件。Google One が7/19に終了します。」
- タップでブリーフィング画面（3）へ

### 3. 今日のブリーフィング（メイン画面）＝ 案A 統合フィード
- 構成: **LLM が書いた要約（上）＋ 緊急順のドメイン別セクション（下）を1画面スクロール**
- 大タイトル = 日付、サブ = あいさつ
- 「今日のまとめ」カード: LLM の日本語プロース要約（締切・GitHub・タスクを1〜2文で）
- セクション（緊急順、iOS グループ化リスト）:
  - **締切が近い**（Canvas iCal + Google カレンダー）: 残日数を赤ピル（例「あと2日」）
  - **GitHub**（各リポジトリの `TODO.md` の LLM サマリー。2026-07-17 変更）: 日本語プロース 2〜3 文。
    TODO の内容が前回と同じなら backend がキャッシュを返す（LLM 呼び出しなし）。
    サマリーが無い旧 payload はリポジトリ別件数にフォールバック。全量は GitHub タブ
  - **要対応**（Gmail トリアージ）: 済ワンタップ＋期限ピル
  - **昨日の GitHub**（gh CLI: commits / PR）: リポジトリ別のコミット数
- フッター: 「07:00 に自動更新 / 更新↻」（プル）
- 表示の作法: リポジトリ名は等幅・開発系は青、締切は赤ピル

### 4. 設定
- 通知: ブリーフィング ON/OFF、配信時刻（既定 7:00 AM）、タイムゾーン（Seattle）、言語（日本語）
- 連携ソース: Google カレンダー / Gmail / Canvas(iCal) の接続状態
- トリアージ: 要対応の判定基準、自動配信メールの除外トグル
- **原則**: マスターデータの編集は持たない。設定は最小限（[spec 2-5](app-features.md)）

## デザイン方針

- **配色**: 夜明けの光をモチーフ。AIの定番（クリーム×テラコッタ）を避け、クールなスレート地 × 琥珀アクセント。締切=赤 / 完了=緑 のセマンティックカラーはアクセントと分離
- **タイポ**: 日本語を含むため Web フォントは使わず **iOS システムスタック（SF Pro / Hiragino）** を意図的に採用。実機と同じ見え方。階層はウェイト・サイズ・数字の等幅（tabular-nums）で作る
- **情報設計**: 要約を上・詳細を下。状態を形で（残日数ピル / 済チェック / 接続バッジ）。締切の逼迫は色で一目化
- **ライト / ダーク両対応**

## この設計から実装 Step 7 への含意

MVP の iOS アプリ（[Step 7](../plans/archive/mvp-morning-briefing.md)）が持つ画面はこの4つ。
バックエンド API との対応:
- オンボーディング → `POST /devices`
- ブリーフィング表示・更新↻ → `GET /briefings/latest`
- 通知本文 → APNs ペイロード（v1 フル内容）
