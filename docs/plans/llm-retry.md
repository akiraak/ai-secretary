# LLM 呼び出しの混雑リトライ（529 対策）

## 目的・背景

本番（g3plus）でブリーフィング手動実行が 2 回連続で Anthropic API の
`529 overloaded_error` により失敗した（収集は正常、LLM 整形段階で落ちる）。
SDK 既定のリトライは 2 回・数秒間隔で、混雑が数分続くと乗り切れない。
毎朝 07:00 の cron が一度の混雑でブリーフィング配信ごと落ちるのは避けたい。

## 対応方針

- `src/llm/briefing.ts` の `messages.create` 呼び出しに外側リトライを追加
  - リトライ対象: 429 / 5xx（529 含む）/ 接続エラー
  - 間隔: 30 秒 → 60 秒 → 120 秒（計 3 回、最悪 +3.5 分。cron ラッパの flock が多重起動を防ぐ）
  - リトライ時は console.warn でログに残す（cron ログで追える）
- スキーマ不一致・4xx（429 以外）は仕様通り即失敗のまま

## 影響範囲

- `src/llm/briefing.ts` のみ

## テスト方針

- `npm run typecheck`
- `npm run llm:check -- --fixture` で通常経路が壊れていないこと
- 本番で管理画面から再実行し、生成 → push まで通ること（混雑が続く場合はリトライログを確認）
